# 시연 가이드 — 따라 하면 되는 데모 진행서 (백엔드 기준)

> 프로젝트를 만들지 않은 사람도 그대로 따라 하면 시연할 수 있게 쓴 진행서입니다.
> 이 디렉터리는 **백엔드**(수집·요약·발송 파이프라인 + 구독자 API)예요. 시연의 핵심인 **메일 발송은 여기서 끝까지 돕니다.** 프론트 화면(구독 신청·관리)은 **별도 프론트 디렉터리**에서 띄웁니다.
> 발표 화면에는 코드를 띄우지 않습니다. 발송은 받은편지함에 도착한 실제 메일로 보여줍니다.

---

## 0. 이 시연에서 보여줄 것

1. **진짜 핵심 — 뉴스레터 메일이 실제로 만들어져 발송된다** (이 백엔드에서) — 관심 키워드로 뉴스를 모아(수집) → AI가 요약하고(요약) → 메일로 만들어(렌더) → 실제 Gmail로 보낸다(발송)
2. (선택) **서비스 화면(프론트)** — 사용자가 구독을 신청하고, 관리자가 구독자를 관리하는 흐름 (프론트 디렉터리에서 실행)

> **왜 `uvicorn`만으로는 부족한가?**
> 구독자 API(uvicorn)는 "구독 신청·관리"용 서버일 뿐이에요. 실제 발송은 그 옆에서 **뉴스 수집(네이버 API) → AI 요약(LLM) → 메일 전송(Gmail SMTP)** 파이프라인이 돌아야 일어납니다. 이 가이드의 시연 B가 바로 그 파이프라인을 보여주는 부분입니다.

---

## 1. 한 번만 하는 준비 (백엔드 폴더에서)

### 1-1. 파이썬과 패키지
Python 3.12 권장.
```bash
pip install -r requirements.txt
```

### 1-2. `.env` 채우기 (가장 중요)
```bash
cp .env.example .env
```
| 환경변수 | 쓰이는 단계 | 발급/설정 |
|---|---|---|
| `NAVER_CLIENT_ID` / `NAVER_CLIENT_SECRET` | 수집 | https://developers.naver.com 앱 등록 후 발급 |
| `OPENAI_API_KEY` | 요약(LLM) | https://openrouter.ai/keys 의 OpenRouter 키 |
| `SENDER` / `GOOGLE_APP_PASSWORD` | 발송 | Gmail 주소 + [Google 앱 비밀번호](https://myaccount.google.com/apppasswords) (2단계 인증 필요) |
| `ADMIN_PASSWORD` | 관리자 화면(프론트) | 원하는 문자열 |
| `API_BASE_URL` | 프론트↔백엔드 | 로컬은 `http://localhost:8000` |

> 시연 B(발송)만 하려면 `NAVER_*` · `OPENAI_API_KEY` · `SENDER` · `GOOGLE_APP_PASSWORD` 만 있으면 됩니다.

---

## 2. 서버 켜기

**백엔드 (이 폴더에서) — 구독자 API**
```bash
uvicorn src.api:app --port 8000
```
> 발송 파이프라인은 서버가 아니라 아래 **시연 B의 `demo_send.py`** 로 돌립니다. (정기 발송 스케줄러는 부록 B의 `main.py`)

**프론트 (프론트 디렉터리에서) — 화면**
```bash
# 예: 프론트 디렉터리(team_project_frontend 등)로 이동해서
streamlit run app.py --server.port 8501
```
> 프론트의 관리자 화면은 그 디렉터리의 `.streamlit/secrets.toml`(`admin_password`)이 이 백엔드 `.env`의 `ADMIN_PASSWORD`와 **같은 값**이어야 합니다(아래 7-3).

## 3. 시연 A — 서비스 화면 (프론트, 선택)

프론트를 띄운 뒤(위 2), 브라우저에서 사이드바로 페이지를 이동하며 진행합니다.

| 순서 | 할 일 | 멘트 |
|---|---|---|
| A-1 | `subscribe`에서 이름·이메일·키워드·받는 시간 입력 후 **구독 신청** | "신청만으로 끝이 아니라, 이메일 확인 링크를 눌러야 발송 대상이 됩니다." |
| A-2 | (받은편지함) 확인 메일 링크 클릭 | "이렇게 이메일 소유를 확인해야 발송됩니다 — 도용 방지." |
| A-3 | `dashboard`에서 관리자 비밀번호 입력 → 목록·통계 | "관리자는 전체 구독자를 여기서 봅니다." |
| A-4 | 구독자 수정·삭제 | "화면에서 바꾼 내용이 곧바로 반영됩니다." |
| A-5 | `user_dashboard`에서 이메일 → 인증 코드 → 조회/수정 | "비밀번호 없이, 이메일로 받은 코드만으로 본인 정보를 다룹니다." |

> 참고: 이메일 주소는 굳이 소리 내 읽지 않아도 돼요. (비밀번호도 아니라 큰일은 아니지만, 신경 쓰이면 — 4번 참고)

## 4. 시연 B — 실제 발송 보여주기

무대에서는 코드를 실행하지 않습니다. **미리 발송해 둔 실물 메일**을 받은편지함에서 열어 보여줍니다.

### 준비 (리허설 때 딱 한 번)
발표에서 열어 볼 주소로 **실물 메일을 하나 받아 둡니다.** → 아래 **부록 A**의 `demo_send.py`를 한 번 실행하면 오늘 날짜로 도착해 있습니다.

### 무대에서
1. (프론트를 함께 쓴다면) 구독 화면에서 "이 주소로 뉴스레터가 갑니다"까지 설명한 뒤,
2. **받은편지함을 열어 도착한 실제 뉴스레터를 보여줍니다.**
   - 멘트: *"구독하면 이런 메일이 실제로 갑니다 — 키워드 뉴스를 AI가 요약해 이렇게 정리해서, 방금 발송된 게 여기 도착해 있죠."*

> 도착한 메일이 "뉴스레터가 어떻게 생겼는지"와 "실제로 배달된다"를 함께 보여줍니다. 미리 발송해 두면 발표 중 네트워크·LLM·SMTP 상황에 영향받지 않습니다.

### 이메일 주소 노출 — 신경 쓰이면 (심각한 건 아님)

이메일 주소는 **비밀번호처럼 민감한 정보가 아니라** 크게 경계할 필요는 없어요. 다만 개인 주소를 굳이 드러내기 싫으면 이렇게 하면 됩니다.

- 주소는 읽지 말고 **가리켜서** — "**이 주소로** 구독하면…", "**방금 그 주소로** 발송된 게 여기 도착했죠".
- 입력할 때도 굳이 읽지 말고 그냥 타이핑하거나 **미리 채워두기**.
- 깔끔하게 하려면 **데모 전용 계정**으로 발신(`SENDER`)·수신 둘 다 쓰면 화면에 개인정보가 아예 안 떠요.
- **녹화·영상으로 공유**할 거면 화면에 남으니, 그때만 데모 계정이나 "받는사람/보낸사람" 크롭 정도 신경 쓰면 됩니다.

## 5. 부록 A — 라이브 발송 (선택 · 시연 보조자가 있을 때)

"지금 보낸다"를 라이브로 보여주고 싶을 때만. **기술 보조자 + 키가 확실히 되는 환경**이 전제입니다.

- 보조자가 **자기 터미널(발표 화면이 아닌 곳)** 에서, 이 백엔드 폴더에서 실행:
  ```bash
  python demo_send.py 받는사람@gmail.com 금리,반도체
  ```
- 이 도구는 **자동으로 한 번에 보내지 않습니다.** 4단계(수집 → 요약 → 렌더 → 발송)를 진행하며 **매 단계 Enter를 눌러야** 넘어가고, **발송 직전에도 한 번 멈춥니다.** → 설명하며 진행하거나 원하는 지점에서 멈출 수 있어요.
- 화면에는 코드가 아니라 사람이 읽는 진행 메시지만 나옵니다:
  ```
  [1/4] 뉴스 수집 중...
        · 금리: 5건
        → 총 10건 수집 완료
    ⏸  계속하려면 Enter...
  [2/4] AI가 요약하는 중...
        → 요약 완료 (총 6개 항목)
    ⏸  계속하려면 Enter...
  [3/4] 메일 형태로 만드는 중...
        → 메일 HTML 생성 완료 (7,980자)
    ⏸  실제로 발송하려면 Enter...
  [4/4] 발송 중...
        ✅ 발송 완료 → 받는사람@gmail.com
  ```
- **발표 화면에는** 이 터미널의 로그만 띄우거나, 마지막에 받은편지함으로 전환합니다. (코드는 절대 띄우지 않습니다.)

> 키가 비어 있으면 발송 전에 어떤 값이 없는지 알려주고 멈춥니다.

## 6. 부록 B — 스케줄러 예약 발송 (참고용)

실제 서비스는 "정해진 시각에 자동 발송"됩니다. 그 원리를 보여주려면 이 폴더에서:
```bash
python main.py
```
→ 스케줄러가 뜨고 매 분마다 발송 대상을 점검합니다. 다만 구독자 발송 시각(30분 단위)이 되어야 나가 **타이밍을 맞춰야 하니**, 라이브보다는 "실제 서비스는 이렇게 예약 발송된다"는 **설명·코드 소개용**으로 권합니다.

## 7. 자주 나는 문제 & 해결

| 증상 | 원인 | 해결 |
|---|---|---|
| 7-1. `demo_send.py`가 "OPENAI_API_KEY … 비어 있습니다" | `.env`에 OpenRouter 키 없음 | `.env`의 `OPENAI_API_KEY` 채우기 |
| 7-2. 발송에서 Gmail 로그인 실패 | 일반 비밀번호를 넣었거나 2단계 인증 미설정 | 구글 **앱 비밀번호** 발급해 `GOOGLE_APP_PASSWORD`에 넣기 |
| 7-3. 관리자 로그인은 되는데 목록이 401 | 백엔드 `.env`의 `ADMIN_PASSWORD` ≠ 프론트 `secrets.toml`의 `admin_password` | 두 값을 똑같이 맞추기 |
| 7-4. 수집이 `0건` | 그 키워드에 최근 뉴스가 없음 | 더 일반적인 키워드로 바꿔 다시 실행 |
| 7-5. 수집에서 네이버 401 | 네이버 키 오타/미설정 | `NAVER_CLIENT_ID` / `NAVER_CLIENT_SECRET` 확인 |
| 7-6. 확인 메일이 안 옴(시연 A) | 발송 설정(`SENDER`/앱 비밀번호) 문제 | 발송 값 확인 — 이게 되면 시연 B도 됩니다 |
